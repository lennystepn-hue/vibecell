# Spec 5B — Portfolio-Intel Design

**Date**: 2026-04-20  
**Status**: Scaffolded (MVP stubs committed)  
**Owner**: vibecell.dev core

---

## Overview

Spec 5B gives vibecell cross-project intelligence. Where Spec 5A tracks individual project health, 5B zooms out to the portfolio level: which projects are alive, which are stagnant, where dependencies are rotting, and what patterns the system can surface to help the builder prioritise.

The core artifact is a `portfolio_snapshot` — a periodic, generated-at-read JSON blob summarising the whole workspace. This snapshot feeds a heatmap, stagnation alerts, dependency rot warnings, and eventually a weekly email brief.

---

## 1. Activity Heatmap

### Design

A grid of projects × weeks, coloured by activity intensity. Inspired by GitHub's contribution graph.

**Data sources** (all existing tables):
- `ships` — `shipped_at` timestamp
- `sessions` — `started_at`
- `decisions` — `decided_at`
- `notes` — `created_at`
- `lifecycle_events` — `occurred_at`

**Aggregation**: for each (project_id, ISO week), count total events. Store as `{week: "2026-W15", project_id: "...", event_count: 7}`.

**Colour scale**: 0 events = grey, 1-2 = light green, 3-5 = medium green, 6+ = bright green. One week = one cell. Rolling 12 weeks shown by default.

### Frontend

`Portfolio.vue` renders a CSS grid:
- Rows: projects (sorted by recent activity)
- Columns: weeks (last 12, oldest left → newest right)
- Cells: coloured `<div>` with `title` tooltip showing event count + week

## 2. Stagnation Detector

### Design

A project is **stagnant** if it has no activity in the last 30 days AND its status is not `archived` or `shipped`. Stagnation is computed at snapshot generation time.

**Stagnation fields in snapshot**:
```json
{
  "stagnant_projects": [
    {"project_id": "...", "slug": "...", "name": "...", "days_since_activity": 45, "last_activity_at": "2026-03-05T..."}
  ]
}
```

**Frontend**: Stagnant projects shown with a warning badge in the heatmap row and a dedicated "Needs attention" section above the grid.

## 3. Dependency Rot

### Design (deferred — Phase 5B.2)

For each project with GitHub repos, call the GitHub Dependabot API:
- `GET /repos/{owner}/{repo}/vulnerability-alerts` — security vulnerabilities
- `GET /repos/{owner}/{repo}/dependabot/alerts` — outdated dep alerts

Aggregate across all repos in the workspace. Surface:
- Count of critical/high/medium alerts per project
- "3 projects use outdated Stripe SDK" cross-project pattern

**Cross-project pattern detection**: scan `project_stack` (StackItem) entries, group by name + version. Flag packages used in multiple projects where a newer version exists.

Deferred to 5B.2 — requires GitHub integration permission expansion.

## 4. Cross-Project Recommendations

### Design (deferred — Phase 5B.3)

Heuristic rules that fire when the snapshot is generated:

| Rule | Trigger | Recommendation |
|------|---------|----------------|
| Shared stack | ≥3 projects use same library | "Extract shared dependency into workspace package?" |
| Multiple Stripe projects | ≥2 projects have `stripe` in stack | "Consider a shared billing service?" |
| Multiple auth projects | ≥2 projects have `auth` in stack | "Shared auth layer opportunity?" |
| Dead project cluster | ≥3 stagnant projects in a group | "Archive or sunset?" |

Recommendations stored in snapshot `recommendations` array. Displayed as callout cards above the heatmap.

## 5. Weekly Brief

### Design (deferred — Phase 5B.4)

Every Monday at 08:00 UTC, generate and email a weekly summary to workspace owners:

**Contents**:
- Ships from last week
- Most active projects
- Stagnant projects (>30d quiet)
- New dependency alerts
- Upcoming domain expiries (from Spec 5A.3)

Generated by `generate_snapshot()` + template rendered via Resend (already wired). Opt-out toggle in Settings.

## 6. Snapshot Architecture

### `portfolio_snapshot` table

Single table, append-only. Latest snapshot per workspace is the one used by the API.

Schema:
- `id` (ULID PK), `workspace_id` (FK → workspaces), `generated_at` (timestamptz), `data` (JSONB)

**Snapshot shape** (abbreviated):
```json
{
  "workspace_id": "...",
  "generated_at": "2026-04-20T10:00:00Z",
  "project_count": 12,
  "active_project_count": 8,
  "stagnant_projects": [...],
  "activity_by_week": [
    {"week": "2026-W15", "project_id": "...", "event_count": 7}
  ],
  "recommendations": [...],
  "dependency_alerts": []
}
```

### Generation trigger

- On-demand: `GET /api/v1/portfolio/snapshot` generates if stale (>1h old) or missing.
- Background: every 15 minutes via APScheduler (Phase 5B.1 impl).
- Manual: CLI `hangar portfolio refresh`.

### Staleness

If the latest snapshot is >1h old, regenerate synchronously. For large workspaces (>50 projects) with >10k events, consider background generation with a "stale" flag returned to the client. Deferred.

## Open Questions

- Should the heatmap show all projects or only the top N most active? (Initial: all, paginate if >30 rows.)
- Weekly brief: opt-in or opt-out? (Opt-out is better for initial engagement — builders should see the value first.)
- Should recommendations be surfaced in MCP? ("You have 3 stagnant projects — want me to archive them?")
- How do we handle multi-workspace users — global portfolio or per-workspace only? (Per-workspace for now.)
